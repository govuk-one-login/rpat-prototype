RPAT — Story Maps

GOV.UK One Login · RP Admin Tool · Last updated April 2026 · Assumes user is signed in via GOV.UK SSO

Key principles informing these maps

  • Terminology: "Client" not "environment" in the UI — technically accurate for the audience (developers/technical staff). RPs can label/tag their clients freely.
  • Agnostic to RP implementation: We don't care why an RP has multiple clients, how their internal environments map, or whether they use shared or per-service OIDC clients (see Les's integration/implementation options diagram). We show whatever clients exist.
  • Three-level IA: Services list → Service overview (prod + integration clients) → Client config detail.
  • Data model: Organisation → Service Integration → OIDC Clients (1 prod, n integration). Service Integration and Organisation are RPAT groupings; OIDC Client is what's in the client registry.
  • Evidence base: DfE (3 services, one with 5+ integration clients), DVLA (1 Service Integration "Identity", multiple clients for Ext/Link/PreProd). Patterns differ — IA must handle both.
  • Slice 1 scope: Integration client creation + editing, read-only production view. Production write, internal admin, RBAC, and promotion flow are later slices (shown faded).
